# Azrael Security Skill — Brand & Communication Standard
**Version:** 1.1
**Date:** 2026-03-23
**Purpose:** Define voice, positioning, and quality standards for all public-facing content produced under the Azrael Security brand.
**Use when:** Darrius asks Claude to help write or review any public-facing content — GitHub READMEs, research writeups, LinkedIn posts, conference abstracts, personal site copy, or any content that represents Azrael Security publicly.

## Brand Positioning

**Azrael Security** is a professional offensive security research brand built on one principle: quality over ease.

The two-line positioning:
- **What:** Red Team Infrastructure Engineer and Vulnerability Researcher
- **How:** Building production-grade offensive infrastructure and conducting systems-level security research on live infrastructure, not synthetic lab VMs

The differentiator that every piece of public content should reflect: the work happens on real infrastructure (Veil, a three-node WireGuard mesh with live C2 and a production honeypot collecting real attacker data) not on CTF platforms or spun-up lab environments. This is what separates the Azrael Security story from the majority of candidates at the same experience level.

## What Azrael Security Is and Is Not

**Is:**
- A professional offensive security research brand operated by a practitioner
- Infrastructure-first: Veil is the proof of operational maturity
- Research-oriented: container boundary weaknesses, RE methodology, CVE analysis
- Practitioner-to-practitioner: the audience is security engineers, not recruiters

**Is not:**
- A generic "cybersecurity" brand
- A CTF portfolio account
- A learning-in-public blog
- A credentials collection

The distinction matters in every content decision. A post about completing a pwn.college dojo is learning-in-public. A post about a specific namespace isolation behavior discovered on live Cerberus infrastructure while reproducing an attack path is research. Only the second belongs under the Azrael Security brand.

## Voice

Write like a practitioner for practitioners. These rules apply to everything published under the Azrael Security name.

**Do:**
- Lead with what you built or found, not what you plan to build or intend to find
- State the mechanism, not just the outcome — "overlayfs exposes host mount points under rootless Podman when X condition is met" not "I found a container escape"
- Use the correct technical term when one exists — namespace, not "container isolation thing"
- Write in past tense for findings and present tense for principles and takeaways
- Let the infrastructure speak — Mythic C2 on NixOS, WireGuard mesh, Cowrie collecting live attacker data are concrete signals, not buzzwords
- Use natural sentence construction — commas and conjunctions over punctuation-as-pause

**Never:**
- Em-dashes anywhere in public-facing copy. No exceptions. Use commas, periods, or restructure the sentence.
- "I am passionate about cybersecurity"
- "I am excited to share"
- "This is a deep dive into"
- "In this post, I will be covering"
- "Stay tuned for more"
- "I am still learning but"
- Inflated maturity claims — if a tool is a proof of concept, say proof of concept
- Hedging that undermines the finding: "This might be interesting to some people"
- Editorializing — do not state the implication, let the facts carry it
- Opening hook sentences that perform a point instead of making one ("Every mesh needs an edge node that doesn't go down" is marketing, not writing)

## AI-Tell Awareness

All public-facing Azrael Security copy must pass an AI-tell check before publishing. The goal is prose that reads as written by a practitioner, not generated by a model.

**What AI-generated text looks like (sources: ECC Faculty Resources, MIT Technology Review):**

**Structural tells:**
- Overuse of bullet points where prose would serve better
- Excessive hedging or over-qualification
- Sentences that are grammatically correct but oddly constructed or overly complex
- Consistent use of common words at higher frequency than natural human writing ("the", "it", "is")
- Lack of typos, idiosyncrasies, or the natural variability of human writing
- Perfect, clean text with no rough edges — human writing is messier

**Voice tells:**
- Editorializing instead of stating — "This is important because..." instead of just stating the fact
- Opening sentences that frame rather than declare — "In this writeup, I explore..."
- Transitions that are too smooth — AI links paragraphs in ways humans wouldn't bother to
- Lack of specific detail in places where a practitioner would naturally be specific
- Generic conclusions that could apply to any writeup on the topic

**The test:** Read the copy aloud. Does it sound like something you would actually say to a practitioner peer? If it sounds like a product description or a LinkedIn post from someone trying to sound professional, rewrite it.

**Practical checks before publishing:**
1. Remove every sentence that states an implication instead of a fact
2. Remove every opening hook that exists to sound good rather than inform
3. Replace em-dashes with commas or restructured sentences
4. Check for over-use of bullet points — if it can be prose, make it prose
5. Read the last sentence of every paragraph — AI often ends paragraphs with a summary restatement. Cut those.

## GitHub README Standard

Every public repo README follows this structure in this order. No reordering, no skipping.

**1. One-line positioning statement**
What the project is and what it demonstrates. Not what it aspires to be.
Example: "Production-grade offensive security homelab infrastructure. Three-node WireGuard mesh, Mythic C2 on NixOS, rootless Podman Quadlets on an 8GB Chromebook edge node."

**2. Architecture or current state**
What exists right now. If it has a diagram, include it. If it has a status table, make the statuses honest — no "planned" rows masquerading as current capability.

**3. What it demonstrates**
The specific technical signals a hiring manager or peer engineer should take from the repo. Written for a practitioner audience, not a recruiter.

**4. Usage or deployment**
How to reproduce or use it. Exact commands, not hand-waving.

**5. Honest status**
What works, what doesn't, what's next. "Nuclei worker not yet deployed" is better than silence or a "coming soon" placeholder.

**What never appears in a README:**
- "Coming soon" sections that have no committed code behind them
- Vague objectives that aren't backed by current state
- CTF rankings or platform scores as primary credentials
- Skill matrices as the lead section
- An objectives statement that reads like a cover letter

## Research Publication Framing

When writing about research, frame it around the infrastructure advantage.

The framing that works:
- "Reproduced on live Cerberus infrastructure (rootless Podman, Arch Linux, kernel 6.x), not a synthetic lab VM"
- "Observed in production traffic from Cowrie honeypot over X days of real attacker interaction"
- "Tested against the Veil WireGuard mesh under real network conditions"

The framing that doesn't:
- "I set up a test environment to reproduce this"
- "In my homelab I found..."
- "While doing CTF challenges I noticed..."

The live infrastructure is the differentiator. Every research publication should make clear that the finding came from real infrastructure with real conditions, not a purpose-built lab that only exists for the writeup.

## Conference Abstract Formula

Every conference abstract follows this structure:

**1. Research question** (one sentence)
The specific question the talk answers. Not a topic, a question.
Example: "Does overlayfs in rootless Podman expose host mount namespaces to a container process under conditions common in red team infrastructure deployments?"

**2. Why it matters** (two to three sentences)
The operational or defensive consequence of the answer. Who cares and why. Connect it to real infrastructure, not theoretical scenarios.

**3. What you found or built** (two to three sentences)
The concrete finding or artifact. Specific, verifiable, not vague. If it is a tool, name it and say what it does. If it is a finding, state the condition and the behavior.

**4. What practitioners take away** (one to two sentences)
The actionable insight. What does someone do differently after seeing this talk?

What never appears in a conference abstract:
- "I will be exploring"
- "This talk will cover"
- "We will look at"
- Scope that cannot be delivered in the time slot
- Findings described so vaguely they could mean anything

## LinkedIn and Social Media Principle

Post about what you built or found. Not about what you are learning, planning, or intending.

**Post when:**
- A research finding is published, link the writeup, state the finding in one sentence
- A significant infrastructure milestone is complete: first Mythic callback, new Veil node integrated, HackSpaceCon talk accepted
- A writeup is committed that demonstrates a specific mechanism, link it, state what mechanism it explains

**Do not post:**
- "I just started learning X"
- "Excited to be working toward Y"
- "Day N of my security journey"
- Course completion announcements as standalone posts
- CTF solve announcements without a linked writeup

**The test for any post:** Would an engineer at runZero read this and learn something concrete, or would they learn only that you exist and are trying? If it is the latter, do not post it.

**Engagement principle:** Commenting on technical posts from HD Moore, Tim Medin, Jorge Orchilles, and their networks with substantive technical additions is higher signal than posting about your own progress. A comment that adds a specific technical detail or asks a precise question about a published finding puts you in the right orbit without requiring a completed research artifact.

## When Claude Is Helping With Public Content

1. Ask what the content is for before writing anything — README, LinkedIn post, abstract, personal site copy. The format rules differ by context.
2. Apply the voice rules before delivering any draft — if a draft contains "excited to share," "passionate about," or an em-dash, rewrite before delivering.
3. For READMEs: ask for the current honest state of the project before writing. Do not fill gaps with aspirational language.
4. For research publications: confirm the finding is reproducible and documented before framing it publicly. A writeup exists in `security-research/` before a LinkedIn post goes out.
5. For conference abstracts: confirm the research question is answerable within the current state of the work. Do not write an abstract for research that does not exist yet.
6. Flag scope inflation — if the draft implies capabilities or completeness that the current state does not support, name it explicitly before delivering.
7. Run the AI-tell check on every draft before delivering. Remove editorializing, opening hooks, em-dashes, and summary restatements at paragraph ends.
